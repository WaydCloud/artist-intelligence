# WORKFLOW — 작업 순서 · 완료 조건 · 리커버리

> 하네스의 "작업순서 / 완료조건 / 실패 시 행동양식". 모든 작업은 이 흐름을 따른다.

## 표준 작업 순서

```
1. SPEC 확인 (무엇을 만드나)
2. RULES 확인 (불변식·규칙)
3. DOMAIN 확인 (도메인 근거)
4. 구현
5. TESTS 수용조건 충족
6. 품질 게이트 (lint · typecheck · smoke · schema-validate)
7. 리뷰
8. 머지
```

## 세로 슬라이스 원칙

가로(모든 백엔드 먼저)로 가지 않는다. **한 모듈을 대시보드 렌더까지 관통**시킨 뒤 같은 패턴으로 복제. 현재 관통 대상: `modules/dance-dynamics`.

## 완료 정의 (Definition of Done)

- [ ] 해당 `SPEC.md`의 완료 정의 충족
- [ ] `RULES.md` 불변식 위반 없음
- [ ] `TESTS.md` 수용조건 통과
- [ ] 모듈 CLI가 **스키마 유효 `report.json`** 산출 (smoke)
- [ ] lint · typecheck · security 통과
- [ ] 관련 문서(SPEC/RULES/DOMAIN 등) 갱신
- [ ] **기준 원장**: 하중받는 지표·임계값이 `RULES.md`에 원장 형식(정의·도메인근거·튜닝·테스트·한계)으로 명시 · 임계값은 튜닝 노출(코드 은닉 금지) — [`AGENTS.md`](AGENTS.md) §2.1

## 리커버리 (실패 시 행동양식)

| 상황 | 행동 |
|---|---|
| CI 실패 | 우회 금지. 로그 → 원인 격리 → 최소 수정 → **재현 픽스처 추가** |
| 스키마 불일치 | 규격 vs 산출 중 무엇이 옳은지 판단 → **문서(규격) 기준으로 정렬**. 스키마 변경 시 대시보드 동시 수정 |
| 핵심 흐름 붕괴 | 즉시 **마지막 green 커밋으로 롤백** → 원인만 격리해 재작업 |
| 도메인/규칙 모호 | 진행 중단 → `DOMAIN`/`RULES` 보강 또는 사용자에게 질문 |

## 브랜치 · 커밋 (git 사용 시)

기능별 브랜치, 작은 커밋. PR에서 게이트 통과 후 머지. 커밋 메시지는 "무엇을·왜".
